研究问题与场景

研究问题

引入知识图谱结构化路径与时态因果建模后，LLM在多跳因果推理中的错误率与幻觉率如何量化下降？

场景：服务器集群故障排查

要素说明
输入告警文本（如「machine-1-3 第 2 维 memory_used_rate 异常」）
任务从历史图谱中检索完整因果链 Component → Symptom → Cause → ... → Solution
输出自然语言答案 + 可解释的多跳因果路径 + 幻觉定位
核心难点因果链跨时态传播、多跳推理易遗忘、LLM容易编造未支持的节点

建模思路（一句话）

为什么需要时态知识图谱？

痛点对比

静态图谱的局限（B3 NoTemporal） 时态图谱的增益（B4 Full Graph-RAG）
只能表达「A caused B」，无法表达「A caused B during 2026-07-09 14:00~16:00」 每条边携带 valid_at / invalid_at，精确记录因果关系的有效时间窗
查询时刻为 15:00 时，已失效的因果边仍然被召回 → 时态不准确 → 路径错误 查询时刻自动过滤：valid_at ≤ query_time ≤ invalid_at
→ 时态准确率低，路径错误率高 → TemporalAccuracy 提升 6.5%，PathErr 降低 16.3%

时态建模的学术依据

  • Zep bi-temporal 模型（valid_at / invalid_at / expired_at）设计依据
  • AAAI 2023 Historical Contrastive Learning for TKG Reasoning
  • GraphRAG-Bench 图构建阶段 Entity/Relation Recall 指标借鉴

系统架构（五模块流水线）

Module 1: Graph Construction (Neo4j + Graphiti)

4类节点: Component / Symptom / Cause / Solution  |  5种关系: HAS_SYMPTOM / CAUSED_BY / ...  |  时态边: valid_at / invalid_at

▼
Module 2: Cypher Query Generation

基于 query 意图分类 (multi_hop_path / causal_chain)  |  模板化 Cypher 生成 (per-hop 不同模板)

▼
Module 3: Temporal Pruning + Path Extraction

时态窗口过滤  |  路径置信度计算（几何平均各跳边置信度）

▼
Module 4: Claim Decomposition + Hallucination Verification

LLM 拆解答案为原子声明  |  LLM 核验每个 claim 是否被图谱路径支撑

▼
Module 5: FastAPI 服务 + G6 前端可视化

答案 + 路径 + claim 核验结果 完整可追溯  |  四层分层布局 + 逐跳高亮动画 + 时间轴滑块

关键发现与诚实说明

发现一：B1/B2 全 0 是设计预期

  • 不是 bug，是 ablation 设计的本意
  • B1 无图基线，B2 仅语义检索，它们的作用是作为「下限」剥离每层的边际价值

发现二：R/AR/EM 全 0 的诚实说明

  • 原因：LLM 生成的 answer 文本不显式包含图谱节点名
  • 不影响对比：4 个 baseline 使用同一 prompt 模板，横向比较公平
  • v3 计划：改 prompt 让 LLM 返回结构化（节点链 + 自然语言）

发现三：confidence=0.860 假象

  • 原因：Neo4j 边的 confidence 字段为空，fallback 到 0.8
  • 不影响对比：4 个 baseline 共用同一默认值
